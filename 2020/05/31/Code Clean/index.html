Chapter 1

  • 每个函数、每个类、每个模块全神贯注于一事，不受四周细节的干扰和污染
  • 尽量少的依赖关系
  • 减少重复代码
  • 对象和方法 功能不宜太多 提高表达力
  • 提早构建简单抽象

Chapter 2 有意义的命名

  • 避免误导
    • 不是list类型 就不要命名为list
  • 有意义的区分
    • accountData account 没区别
    • customerInfo customer 没区别
  • 使用能读的名称
    • 反例：genymdhms 生成 年 月 日 时 分 秒。。。
  • 使用可搜索的名称（参考：名字长短与作用域大小对应）
  • 类名 对象名：名词/名次短语
  • 方法名：动词/动词短语
    • 参考 get set is等前缀
  • 一以贯之的命名法 统一格式
  • 添加有意义语境
    • 添加统一前缀

Chapter 3 函数

  • 短小 每个函数都只做一件事，依序把你带到下一个函数中
  • 只做一件事
  • 自顶向下读代码
  • 使用具有描述性的名称
    • 同一模块的函数 命名方式保持一致
  • 参数 尽可能少的参数
  • DRY(Don’t Repeat Yourself)
  • 一开始的函数都是冗长复杂的，需要花时间再次分解 优化
  • 编程艺术 将系统的股数

Chapter 4 注释
若编程语言具有足够的表达力，那也许并不需要注释

  • 尽量用代码表达 而非注释
  • 注释不能美化糟糕的代码
  • 好的注释
    • 法律信息
    • 对意图的解释
    • 警示 警告其他程序员会出现某种后果的注释
    • TODO 程序员认为应该做 但是没有做的工作
    • 放大 解释不合理之物的重要性
    • 公共api的描述 注意不要误导 提供准确信息
