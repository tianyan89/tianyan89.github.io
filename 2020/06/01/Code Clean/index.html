Code Clean

2020-06-01
笔记

《代码整洁之道》作者: [美]Robert C. Martin
https://book.douban.com/subject/4199741/

Chapter 1

  • 每个函数、每个类、每个模块全神贯注于一事，不受四周细节的干扰和污染
  • 尽量少的依赖关系
  • 减少重复代码
  • 对象和方法 功能不宜太多 提高表达力
  • 提早构建简单抽象

Chapter 2 有意义的命名

  • 避免误导
    • 不是list类型 就不要命名为list
  • 有意义的区分
    • accountData account 没区别
    • customerInfo customer 没区别
  • 使用能读的名称
    • 反例：genymdhms 生成 年 月 日 时 分 秒。。。
  • 使用可搜索的名称（参考：名字长短与作用域大小对应）
  • 类名 对象名：名词/名次短语
  • 方法名：动词/动词短语
    • 参考 get set is等前缀
  • 一以贯之的命名法 统一格式
  • 添加有意义语境
    • 添加统一前缀

Chapter 3 函数

  • 短小 每个函数都只做一件事，依序把你带到下一个函数中
  • 只做一件事
  • 自顶向下读代码
  • 使用具有描述性的名称
    • 同一模块的函数 命名方式保持一致
  • 参数 尽可能少的参数
  • DRY(Don’t Repeat Yourself)
  • 一开始的函数都是冗长复杂的，需要花时间再次分解 优化
  • 编程可以是艺术

Chapter 4 注释
若编程语言具有足够的表达力，那也许并不需要注释

  • 尽量用代码表达 而非注释
  • 注释不能美化糟糕的代码
  • 好的注释
    • 法律信息
    • 对意图的解释
    • 警示 警告其他程序员会出现某种后果的注释
    • TODO 程序员认为应该做 但是没有做的工作
    • 放大 解释不合理之物的重要性
    • 公共api的描述 注意不要误导 提供准确信息
  • 不好的注释
    • 误导性注释
    • 废话注释
    • 用整理代码的决心替代写废话
    • 能用函数或变量时 就别用注释
    • 删除位置标记
    • 注释的代码要删除！！
    • 无关的信息
    • 为短函数取个容易理解的名字比注释好

Chapter 5 格式
编写的功能极有可能在下一版本中被修改，但是代码风格和可读性仍会影响到可维护性和可扩展性

  • 短文件比长文件更易于理解
  • 空白行标识出独立的概念
  • 代码从上到下 从概述到细节
  • 紧密相关的代码应该互相靠近
  • 避免在不同的文件中跳来跳去，关系密切的概念应放在同一文件下
  • 相关函数，若函数中调用另一函数，调用者尽可能放上面 被调用函数应尽可能放下面 自顶向下，增强可读性
  • 在大家都知道的地方放置常量
  • 概念相关，拥有共同命名方式的代码应放在一起
  • 一行代码不要太长
  • 空格字符
    • 关系相近不加空格，需要强调的加空格
    • 赋值操作 a = b 强调目的，加强分隔效果
    • function test(a, b) {} 突出参数用空格隔开，函数名与参数括号关系相近所以不加
    • a*a + b*b

Chapter 6 对象和数据结构

  • 过程式代码易于添加新函数，面向对象的代码易于添加新类
